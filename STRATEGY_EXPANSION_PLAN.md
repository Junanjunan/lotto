# Lotto 확장 기획: 더 많은 전략 + 더 많은 기능

최종 목적
- 사용자가 “숫자 생성기”만 쓰는 수준에서 벗어나, **전략 실험·통계 탐색·결과 재활용**까지 이어질 수 있게 개선
- 기존 구조(`app/services/strategies.py`, `app/main.py`, `app/static/index.html`)를 유지하면서 확장
- 광고/기타 기능과 충돌 없이 독립적으로 적용 가능

## 1) 현재 상태 요약 (레포 기준)
- 전략은 현재 2개
  - `low_overlap_random`
  - `uniform_random`
- 선택 옵션은 8개 (`avoid_birthday_bias` 등)로 제약형 생성
- `/meta`는 전략/옵션 라벨만 내려주고, UI는 하드코딩 라디오/체크박스로 렌더링
- 평가 지표는 `1~5등` 분포와 `hits` 목록 저장
- 게임 히스토리 API는 기본 메타/세트 목록만 제공

**개선 여지**
- 전략 수 부족 (2개)
- 전략 조합성이 낮음(옵션이 생성 함수 외부에서만 제약)
- 통계 뷰가 제한적: 사용자가 과거 패턴을 탐색·비교하기 어렵다

## 2) 사용자 선호도가 높은 확장 포인트
### A. “전략”은 더 이상 버튼만 늘리는 방식이 아니라 “룰셋”으로
1. **과거 빈도 가중 전략 (Hot/Cold)**
   - 아이디어: 최근 N회차 기준 각 번호의 출현 빈도 계산
     - `hot`: 빈도 높음 가중치 증가
     - `cold`: 빈도 낮음 가중치 증가
     - `balanced`: 고빈도/저빈도 비율 균형
   - 구현 난이도: 중간
   - 기대 반응: “유저가 스스로 난이도/성향을 고르는 느낌” 높음

2. **구간 분할 전략 (Zone Balanced Strategy)**
   - 1~45를 `1~10, 11~20, ...` 구간으로 나누고 각 구간별 출현 수를 제한
   - 예: `(1개-2개) / 구간` 조합, 혹은 중앙값 기반 제약
   - 구현 난이도: 낮음
   - 기대 반응: 현재 `balanced_high_low`의 단순 버전에서 확장된 형태

3. **연속성 제어 전략 (연속수/거리 기반)**
   - 현재는 연속 1쌍 포함/미포함 제어가 없음(현재는 include만 존재)
   - 확장: 정확히 0~2쌍, 1~3쌍, 최소 간격 보장(최소 거리 2 이상)
   - 구현 난이도: 낮음
   - 기대 반응: 패턴 통제 가능

4. **합/범위 제약 전략 (sum / span strategy)**
   - 단순 합 범위(100~170) 외에
     - 극단 수치 제외(합이 너무 낮거나 높은 조합 축소)
     - `max-min` 범위를 제한해 극단 분산 줄임
   - 구현 난이도: 낮음

5. **보너스 동반 전략 (Bonus-aware)**
   - 보너스 포함 가능성 기반 필터를 옵션으로 제공
   - 예: 보너스는 최대 0~1개 포함
   - 구현 난이도: 중간

6. **회귀(미출현) 전략 (Lag Strategy)**
   - 마지막 N회차 미출현 횟수가 긴 숫자 위주로 가중치 부여
   - 구현 난이도: 중간~높음(최근 구간 계산 필요)
   - 구현 포인트: 최근 N회차 기준 계산 캐시

7. **반복 억제 강화 전략 (Portfolio 2.0)**
   - 기존 `low_overlap_random` 개선형
   - 게임 생성 이력을 고려해 특정 번호가 초반 게임에서 과도 점유한 경우 감점
   - 현재 `usage` 기반 로직 확장 버전

### B. 전략보다 더 큰 “무언가”
1. **동시 전략 비교 모드**
   - 3개 전략을 한 번에 생성해 비교 표시
   - 점수형 지표: 총 히트, 기대 1등/2등 확률(분포 가중치), 분산 점수
   - 구현 난이도: 중간(백엔드 응답 포맷 확장 + UI 테이블)

2. **과거 추첨 통계 대시보드 탭 추가**
   - 번호 빈도 Top/Bottom, 보너스 빈도, 구간 빈도, 마지막 출현 간격 히스토그램
   - 구현 난이도: 낮음~중간(SQL 집계 + 렌더링)

3. **개인 프리셋/필터 저장**
   - 자주 쓰는 옵션 조합 저장(로컬저장소 기반)
   - 게임생성 전 `로또 조합 템플릿` 불러오기
   - 구현 난이도: 낮음(localStorage)

4. **추천 조합 북마크**
   - 생성 결과에서 “관심 조합” 저장, 재평가, 복제/재생성
   - 구현 난이도: 중간(DB 저장 or localStorage)

5. **전략 설명 모듈 보강 (설명 품질)**
   - 단순 텍스트를 `도입 이유 / 피해야 할 오해 / 예시` 패턴으로 통일
   - 구현 난이도: 낮음

## 4) 적용 방법 (코드 기준)
### 4.1 백엔드(권장 수정 파일)
- `app/services/strategies.py`
  - `STRATEGIES`에 새 전략 함수 추가
  - 제약 옵션(`Strategy` 옵션) 확장
  - 빈도/구간/연속성 계산은 입력 draw를 받아 사전 계산 후 가중치 계산
- `app/schemas.py`
  - `StrategyOptions`에 새 옵션 키 추가
  - 장기적으로는 옵션을 `strategy_params: dict[str, bool|int|float]` + validation 보조로 확장 고려
- `app/main.py`
  - `/meta`에 전략 목록/옵션 상세/기본 파라미터 반영
  - 필요 시 `/api/stats/lottery` 신규 엔드포인트 추가(빈도/구간/갭 분석)
- `tests/test_strategies.py`
  - 새 전략 생성 수량/고정값 재현성/제약 충족성 테스트 추가

### 4.2 프론트엔드
- `app/static/index.html`
  - 하드코딩 전략 목록을 `/meta` 기반 동적 렌더링으로 전환 검토
  - 전략별 파라미터 UI를 `meta`의 `strategy_option_schema` 형태로 구성(선택형/범위형/토글)
  - 비교 모드: 여러 전략 결과를 탭/카드로 표시
  - 통계 탭 추가: 최근 출현 빈도, 홀짝/구간 히스토그램

## 5) 즉시 반영 가능한 “전략 후보” 6개
아래는 구현 난이도와 부작용이 낮은 순으로 우선순위 제안

1. `zone_balance` (구간 균형)
2. `consecutive_pair_count` (연속수 개수 0~2 제어)
3. `span_guard` (max-min 범위 제어)
4. `sum_targeted` (sum/band 다중 범위 옵션)
5. `cold_weight` (최근 미출현 숫자 가중)
6. `hot_cold_mix` (고빈도/저빈도 비율 조정)

> 위 6개는 기존 `STRATEGY_OPTIONS` 구조를 크게 변경하지 않고도 단계적으로 추가 가능

## 6) 제안 API 스펙 (예시)
- `GET /api/meta`
  - `strategy_options`를 단순 라벨맵에서
    ```json
    {
      "strategy_options": {
        "avoid_birthday_bias": {"label":"...", "type":"bool", "default": false},
        "zone_balance": {"label":"구간 균형", "type":"int", "min":0, "max":4, "default":2}
      }
    }
    ```
    형태로 확장
- `POST /api/games`
  - `options`에 새 키 추가
- 신규 `GET /api/stats/lotto`
  - `number_frequency`, `zone_frequency`, `sum_distribution`, `gap_distribution`

## 7) 품질/리스크 체크
- 무작위 제약 과잉 시 생성 실패 가능성 큼
  - 재시도 상한/백오프 + 제약 완화 fallback 전략 필요
- 사용자가 “난수가 전부”라는 인식이 생길 수 있으므로 전략 설명과 실험 기록 UI를 같이 제공
- 과적합 위험
  - “과거 데이터에 맞춘 전략”은 변동성이 큼 → 정확도 과장 표현 방지 문구 필요
- DB 용량
  - 비교 모드/통계 탭 확대 시 과거 전체 hits 조인 비용 증가 가능 → 캐시 전략 필요
